自我介紹與求職動機

能請你簡單自我介紹嗎？（完整版 / 精簡版）

完整版（約 1 分鐘）

您好，我是一名資深前端工程師，目前任職於一間科技公司，累積近 10 年前端開發經驗。技術主軸是 Vue 3、React Hooks 與 TypeScript，擅長把複雜的業務邏輯拆解成可維護的前端架構，也常在跨團隊協作中扮演技術整合與溝通的橋樑。

現職主要參與智能車隊管理系統的前端建立，主導 Vue 3 (Pinia) 核心開發，並在 Monorepo 架構下與 NestJS 後端共構；我們以 ts-rest + Zod 推行 Contract-first API，把前後端型別契約收斂到單一來源——後端 schema 一改動，前端在編譯期就會被攔截，大幅降低串接溝通成本與 runtime 錯誤。另外我也獨立負責 AMR/AGV 無人搬運車多階段 POC 的戶外定位校正介面與導航 UI。

在這之前我在 前一間公司待了 7 年，主導多個內部後台從 0 到 1，經歷完整的技術棧演進：從早期 React + Redux + Ant Design 3，一路推進到 React Hooks、TypeScript、Ant Design Pro、Tailwind，並導入 Jest、React Query + Zod 建立型別契約與測試基礎。

作為資深工程師，我很重視團隊的技術升級與知識累積：會定期帶新技術導讀、輔導初階工程師釐清任務切分與技術選型，最近也把 AI 輔助開發工具（Claude Code）導入團隊日常，並把機器自走車教學手冊重構成以操作流程為主軸的結構化技術文件。未來希望往前端架構師／Tech Lead 發展，並補強 WebGL／Three.js 在 3D 場景（自動化倉儲、車隊路徑視覺化）的應用。

您好，我是一名資深前端工程師，近 10 年經驗，主軸是 Vue 3、React 與 TypeScript。目前在現職公司主導一套智能車隊管理系統的前端，用 Vue 3 + Pinia 開發，並在 Monorepo 下以 ts-rest + Zod 跟 NestJS 後端做 Contract-first 串接，也負責 AMR/AGV 無人搬運車的定位校正與導航 UI。先前在前一間公司任職 7 年，主導多個後台從 0 到 1 並推動技術棧現代化。我擅長把複雜邏輯拆成可維護的架構，也習慣用型別系統與測試把錯誤擋在前面。

你的求職動機是什麼？需替換公司

我的求職動機主要有兩個方面：第一，我希望加入一個制度完善、開發流程成熟的團隊，能在穩定且有節奏的環境裡發揮專長、持續成長；第二，我很重視能不能持續接觸並掌握新技術。在研究貴公司的產品與技術棧後，我發現有不少地方跟我累積的方向（型別系統、Monorepo、AI 輔助開發、3D 視覺化）很契合，也有我想深入學習的部分，這讓我很期待能加入。

面試提醒：務必加入「對該公司的具體觀察」——產品、技術部落格、團隊文化、近期專案等，越具體越顯誠意。
選擇就職公司時，前三項考量依序是什麼？

依序是：1. 團隊氛圍／友善的工作環境、2. 公司制度、3. 開發流程。

對我來說，好的團隊氛圍能讓技術討論與 code review 良性運作；完善的制度讓人能安心長期投入；成熟的開發流程（需求拆分、測試、CI/CD）則直接影響交付品質與成長速度。

你對三年後有什麼職涯規劃？未來想走技術職還是主管職？

未來 2~3 年，我希望往前端架構師／Tech Lead 方向發展，深化大型專案的模組化、型別系統與跨團隊協作能力。技術上計畫補強 WebGL／Three.js 在 3D 場景的應用（對應自動化倉儲、車隊路徑視覺化），並持續深化 AI Agent 工具在開發流程中的整合。

路線上我偏向「以技術為底、再承接團隊責任」——希望在資深專業之上，同時兼顧團隊培育與架構規劃，產出能長期維運的前端基礎建設。

想成為主管嗎？

我對成為主管持開放態度，特別是如果這能讓我更好地支持團隊、提升整體專案的品質與效率。我在技術層面已累積足夠經驗，也能協助團隊解決複雜問題；同時我也清楚，管理不只是技術上的領導，更是關於如何激勵與支持團隊成員、營造良好的工作氛圍。我願意持續學習與成長，以便將來能更好地承擔這樣的責任。

你期望這份工作／這家公司能給你什麼？什麼情況下會讓你想離職？

期望：能持續接觸新技術與有挑戰的專案、清楚的成長路徑，以及尊重專業、能良性討論的團隊；若有國際化／跨團隊合作的機會更好。

會讓我考慮離開的情況：長期缺乏成長空間、技術債無法被正視、或團隊溝通與制度持續無法改善。我習慣先主動溝通、嘗試推動改善，把離職當成最後選項。

對這份工作有什麼期望或規劃？

短期希望快速融入團隊與既有架構，把手上的功能穩定交付；中期希望能在型別系統、Monorepo 與測試流程上貢獻經驗，協助團隊把開發基底打穩；長期則期望有機會參與架構決策，並爭取國際化／跨團隊合作的機會。

假如錄取後偶爾需要你支援其他任務，你會排斥嗎？

不會排斥。我本來就習慣跨域協作——目前工作就需要與硬體／韌體團隊對接，也曾擔任團建負責人、主動把 AI 工具導入團隊。只要任務有助於專案或團隊，我很樂意支援；唯一會留意的是排序與時程，會先跟 PM 確認優先級，避免影響本來該交付的關鍵工作。

經歷與行為面試題

過去在學習前端上有遇到什麼挫折嗎？怎麼克服的？

有兩個比較深刻的：

  • 前期沒自信、容易鑽牛角尖：剛當資深時，常擔心自己的判斷是否正確。後來透過固定的 code review 與技術分享會建立自信——把想法攤開來讓大家檢視，反而更快確認方向；同時靠單元測試多做驗證，讓「對不對」有客觀依據，而不是憑感覺焦慮。
  • 導入 TypeScript 的陣痛：在某代理平台後台第一次替團隊導入 TypeScript + Jest 時，初期型別報錯多、團隊抗拒。我的做法是漸進式導入（先 any 過渡、再逐步收斂型別）、把常見型別寫成範例與文件、並在分享會帶大家實作，最後成為後續多個專案沿用的基底。
請分享你做過技術上最困難的專案？

有兩個代表性的：

1. CMS 後台重構（多語系 + 多層級）：接手一個結構複雜、含多語系與多層級權限的後台。難點在於既有狀態管理混亂、重複請求多。我評估 SWR / React Query 後選用 ahooks 強化快取與狀態，並逐步把舊邏輯抽成共用元件，採「新舊並存、慢慢全轉」的策略，最後不只完成重構，也把先前積壓的需求一併補完。

2. 智能車隊系統的 Contract-first 串接：在 Monorepo 下與 NestJS 後端共構，用 ts-rest + Zod 把前後端型別契約收斂到單一來源。難點在於前後端要同步迭代、且 AMR/AGV 的定位資料格式常變動；Contract-first 讓 schema 改動能在編譯期被攔截，大幅減少串接錯誤。

最有成就感的專案？

接手別人留下的專案，協調「新舊先共用、再慢慢全轉成新的」，過程中還把之前積壓的需求都一併做完。最有成就感的是——不只把技術債清掉，也讓團隊後續開發更順、新人能依共用元件快速上手，等於把短期交付和長期維運同時顧到。

🎤 口頭講稿（約 35 秒）最有成就感的，是一個我接手別人留下的後台專案。當時程式碼結構很亂、又積了不少還沒做的需求。我沒有選擇直接砍掉重寫，而是用「新舊先共用、再慢慢全轉」的策略——先把核心邏輯抽成共用元件，讓新舊版本可以並存，再分批切換。最後不只把技術債清掉、積壓的需求也都補完，團隊後續開發變順了，新人可以直接照共用元件上手。對我來說最有成就感的點，是同時顧到了短期交付跟長期維運，而不是只解一邊。
專案的難點跟亮點是什麼？

難點：

  • 跨硬體／韌體團隊協作，純軟與軟硬之間的對焦成本高、問題定位需要來回驗證。
  • AMR/AGV 戶外定位校正需要貼近實機操作邏輯，測試環境取得不易。
  • 前期公司流程不夠成熟（如部分專案缺 CI/CD、正式機接觸受限、後端技術不統一）。

亮點：

  • 有很多共用元件要從 0 設計，可以把架構與規範一次打好。
  • 導入 Monorepo + ts-rest + Zod 的 Contract-first，前後端可同步迭代。
  • 把 AI 工具帶進團隊日常，並重構技術文件，提升跨部門交付效率。
請分享一個你過去半年改善過的事？（以改善溝通／流程為例）

我參與建立屬於公司的專案模板（boilerplate），把常用的目錄結構、ESLint/Prettier/Husky、型別契約寫法、共用 hook 與元件收斂進去；新專案 setup 時間明顯縮短，團隊寫法也更一致。另外我把機器自走車教學手冊重構成以操作流程為主軸的結構化技術文件，讓跨部門交付的資訊一致、新進工程師能依文件自行演練。

請分享一次你與團隊方向意見相左的經驗，你如何化解衝突？

CMS 後台重構時，我和 PM 對「是否一次性重寫 vs. 漸進式重構」有不同看法。PM 希望快速看到成果，我則擔心一次性重寫風險高、會擋住既有需求。

我的做法是用資料和折衷方案溝通：先盤點風險與工時，提出「新舊並存、先共用核心元件、分批切換」的路線，並把里程碑拆細讓 PM 能持續看到進度。最後雙方都接受這個平衡點，既控制了風險，也維持了交付節奏。

🎤 口頭講稿（約 35 秒）印象最深的是 CMS 後台重構那次，我跟 PM 的方向不太一樣。PM 希望一次性重寫、快點看到成果，我則擔心一次重寫風險太高、又會擋住正在進行的需求。我沒有硬碰硬，而是用資料溝通——先盤點重寫的風險跟工時，再提出一個折衷方案：新舊並存、先把核心元件共用化、再分批切換，並把里程碑拆細，讓 PM 每個 sprint 都看得到進度。最後我們就用這個方案，既控制了風險，也維持了交付節奏。我覺得化解衝突的關鍵是對事不對人，用「目標加可行方案」取代單純反對。
過往開發上遇到不合理的需求如何處理？

我會先釐清需求背後的目標（對方真正想解決什麼），再向對方說明不合理之處——例如時程不可行、會犧牲體驗或品質、或與既有架構衝突。接著提出替代方案與取捨，協調出雙方都能接受的平衡點。重點是對事不對人，用「目標 + 可行方案」取代單純拒絕。

開發上遇過的困難，如何解決？

我的解題順序通常是：1. 自己先重現問題、縮小範圍；2. 與團隊討論、做 pair；3. 查官方文件與社群（issue、討論串）；4. 現在也會善用 AI 工具輔助（例如釐清 TypeScript 型別錯誤、梳理複雜邏輯）。找到解法後會回頭補上測試或文件，避免同樣的坑再踩一次。

請分享一次你犯錯的經驗？

可挑一個「影響可控、有學到東西」的例子，例如：曾在改動共用元件時沒考慮到其他頁面也在使用，導致連帶 bug。

處理方式：第一時間誠實回報、快速 rollback 止血，再補上回歸測試與型別約束，並在分享會提醒大家「動共用程式碼前先確認使用範圍」。我從中學到：共用程式碼的改動要先盤點 caller，並用測試把風險擋在前面。

面試提醒：選錯誤要「真誠但收斂」——重點放在你如何止血、修正與避免再犯，而非錯誤本身有多嚴重。
🎤 口頭講稿（約 35 秒）我分享一個影響還算可控、但讓我學到東西的例子。有一次我改一個共用元件，只想著我負責的那一頁，沒注意到其他頁面也在用，結果上線後連帶出現 bug。我第一時間就誠實回報、先 rollback 把影響止住，再補上回歸測試跟型別約束才重新上線。後來我也在分享會提醒大家，動共用程式碼之前一定要先盤點有哪些地方在用。這件事讓我養成一個習慣：改公用的東西之前先確認 caller，並且用測試把風險擋在前面。
請分享一次你做超出職位範圍事情的經驗？

兩個例子：1. 我擔任過團建負責人，主動規劃活動、凝聚團隊；2. 我主動把 AI 輔助開發工具（Claude Code）導入團隊日常，並重構技術文件——這些都不在原本的職務描述裡，但對團隊效率有實際幫助。我認為資深工程師的價值之一，就是看到能讓團隊更好的事，會主動承接。

目前團隊中有什麼覺得可以敬佩的對象嗎？

團隊裡有位同仁很會做技術分享——同樣的內容，他能講得有結構又吸引人，讓聽眾很容易吸收。我很欣賞這點，也希望自己未來在報告與表達上能做到一樣有感染力。

形容一下你自己 / 別人都怎麼形容你？

工作上：別人常說我是「非常有責任感」的人，交付的東西會顧到品質與後續維運。

個人上：我接觸的事物很多、興趣廣，算是個有趣、好相處的人，在團隊裡容易跟不同角色合作。

你的優缺點是什麼？

優點：情緒穩定、擅長團隊合作、喜歡學習新知，能在跨團隊間扮演溝通橋樑。

缺點：前期比較沒自信、容易鑽牛角尖。後來我靠code review 與技術分享會建立自信，並用測試多做驗證，讓判斷有客觀依據——現在能更快地下決定並對外溝通。

面試提醒：缺點要搭配「已採取的改善行動」，展現自我覺察與成長。
過往專案的團隊人數、角色，自己擔任什麼角色？

團隊約 9 人，組成包含 PM、後端、QA、UI／企劃與前端。我擔任資深前端，除了核心開發外，也常扮演「中間人」——對接企劃需求、協調前後端介面、並輔導初階工程師。現職還多了與硬體／韌體團隊的溝通橋樑角色。

有沒有與設計師合作的經驗？

因為近幾年多在做內部後台系統，直接與專職設計師合作的機會相對較少（多以 Axure／ClickUp 對齊 UI、企劃出規格）。不過我對與設計師合作很有興趣，也熟悉 Figma／Zeplin 的交付流程，很期待能在更重視設計的產品上深入協作。

團隊開發經驗？

我有完整的 Vue 與 React 雙棧團隊經驗：早期用 Vue 開發多語系電商與區塊鏈手機版產品，後來在前一間公司主導多個 React 後台，現職則回到 Vue 3 (Pinia) 主導車隊系統前端，並與 NestJS 後端、硬韌體團隊協作。我習慣以 Scrum 運作，重視介面契約與共用元件的建立。

過往如何學習一個新技術？
  • 加入各大技術社群、追蹤新技術動態。
  • 先看影片／官方文件建立整體概念。
  • 有時間就動手做小專案實作，把概念落地。
  • 現在也會用 AI 工具加速理解與排錯。

例如最近學 Three.js，就是從基礎 → React Three Fiber (R3F) → Drei 一路實作上來。

開發流程與團隊協作

通常你的開發流程是什麼？（Scrum）

Vue：watch 和 watchEffect 的差異？
  • 偵聽來源：watch 需明確指定要偵聽的來源（ref / getter / 多個來源）；watchEffect 自動收集回呼裡用到的響應式依賴。
  • 執行時機：watch 惰性（lazy），預設來源變動才執行（可加 immediate: true）；watchEffect 立即執行一次（eager），之後依賴變動再跑。
  • 新舊值：watch 回呼能拿到 (newVal, oldVal)；watchEffect 拿不到舊值。
  • 深度偵聽：watch 可設 deep: true。

選用：需要新舊值比較、偵聽特定來源、或要惰性執行 → watch；副作用依賴多個值、想自動追蹤、初始就要跑一次 → watchEffect。

// watch：指定來源、拿得到舊值
watch(count, (n, o) => console.log(n, o));
// watchEffect：自動追蹤、立即執行
watchEffect(() => console.log(count.value, name.value));
🎤 口頭講稿（約 30 秒）兩個都是偵聽響應式資料，差別主要在三點。watch 要我自己指定偵聽的來源，預設是惰性的、資料變了才跑，而且回呼裡拿得到新值跟舊值；watchEffect 則會自動收集回呼裡用到的依賴，一開始就立即執行一次，但拿不到舊值。所以如果我需要比較新舊值、或只想針對特定來源做事，我會用 watch；如果副作用同時依賴好幾個值、又想初始就跑一次，用 watchEffect 比較省事。
Vue：父子元件的生命週期執行順序？

口訣：父先開始、子先完成；父最後才 mounted。

  • 掛載：父 beforeCreate → 父 created → 父 beforeMount → 子 beforeCreate → 子 created → 子 beforeMount → 子 mounted → 父 mounted。
  • 更新：父 beforeUpdate → 子 beforeUpdate → 子 updated → 父 updated。
  • 卸載：父 beforeUnmount → 子 beforeUnmount → 子 unmounted → 父 unmounted。

原因：父要渲染時遇到子元件，會先把子元件整個掛載完，才能完成自己的掛載。
（Vue 3 用 beforeUnmount/unmounted，Vue 2 是 beforeDestroy/destroyed；資料請求放 created 或 mounted 皆可，需操作 DOM 才放 mounted。）

🎤 口頭講稿（約 30 秒）順序的口訣是「父先開始、子先完成，父最後才 mounted」。掛載的時候，父元件會先走到 beforeMount，遇到子元件就先把子元件整個建立、掛載完，等子的 mounted 跑完之後，父的 mounted 才會觸發。更新跟卸載也是類似——父先進入、但子先結束。實務上我會把需要操作 DOM 的初始化放在 mounted，純資料請求放 created 或 mounted 都可以。
前端記憶體洩漏：常見成因與怎麼排查？

常見成因：

  • 未清除的 setInterval / setTimeout。
  • 未解綁的事件監聽（addEventListener 沒 remove）、未取消的訂閱（EventBus、RxJS、WebSocket）。
  • Detached DOM：元素從畫面移除，但 JS 變數仍引用著它。
  • 閉包長期持有大物件；全域變數 / 無上限的快取。
  • 元件卸載沒清理（React 沒在 useEffect return cleanup；Vue 沒在 onUnmounted 清）。

排查方式（Chrome DevTools）：

  • Memory → Heap snapshot：操作前後各拍一張，用 Comparison 看哪類物件持續增長、找 Detached 節點。
  • Performance / Memory timeline：反覆操作，看記憶體曲線是否只升不降（正常應在 GC 後回落）。
  • Allocation instrumentation on timeline：定位是哪段程式碼一直配置記憶體。
🎤 口頭講稿（約 30 秒）最常見的成因大概有幾種：沒清掉的計時器、沒解綁的事件監聽或訂閱，還有 detached DOM——就是元素從畫面移除了，但 JS 還抓著它的引用。所以我在 React 會用 useEffect 的 cleanup、Vue 在 onUnmounted 裡把這些清掉。排查的話我會用 Chrome DevTools 的 Memory 面板，操作前後各拍一張 heap snapshot 做比對，看哪類物件一直長、有沒有 detached 節點；或看 memory timeline 的曲線是不是只升不降、GC 之後沒回落。
虛擬捲動（Virtual Scrolling）是什麼？怎麼實作？

面對上萬筆的長列表，若全部渲染會產生大量 DOM 造成卡頓。虛擬捲動只渲染可視範圍 + 上下緩衝（buffer）的項目，捲動時動態替換內容。

關鍵實作：

  • 外層容器固定高度 + overflow:auto；用一個與「總高度」等高的撐高元素，讓捲軸比例正確。
  • 監聽 scrollTop，依每項高度算出 startIndex / endIndex，只渲染這段。
  • 用 transform: translateY() 或 padding-top 把可視區塊推到正確位置。
  • 等高項目最好算；動態高度需量測或估算 + 快取每項高度。

常用套件：vue-virtual-scroller、react-window / react-virtualized、TanStack Virtual。

🎤 口頭講稿（約 30 秒）虛擬捲動是處理上萬筆長列表的方法。如果整包都渲染，DOM 太多會很卡，所以我只渲染可視範圍加上下緩衝那幾筆，其他的用一個跟總高度一樣高的元素把捲軸撐出來。捲動的時候監聽 scrollTop，算出現在該顯示哪一段，再用 transform 把它推到正確位置。等高的項目最好算，動態高度就需要量測或估算再快取。實務上我會直接用 vue-virtual-scroller 或 TanStack Virtual。
Canvas 的渲染流程？跟 DOM 渲染差在哪？

Canvas 是即時模式（immediate mode）的點陣繪圖：

  1. 取得 context（getContext('2d') 或 WebGL）。
  2. 每一幀先 clearRect 清空畫布。
  3. 重新計算並逐一繪製所有圖元（路徑、文字、圖片）。
  4. 用 requestAnimationFrame 排下一幀。

與 DOM 的差異：

  • DOM 是保留模式：有節點樹、瀏覽器自動重繪、自帶事件系統。
  • Canvas 畫完就是像素、沒有物件樹；互動要自己做命中測試（hit testing），文字排版、狀態也都得自己管。
  • 效能：大量圖元（表格、圖表、地圖、遊戲）用 Canvas 遠勝 DOM——例如表格庫用 Canvas 渲染，上萬列也只畫可視區、自繪格線與文字。
🎤 口頭講稿（約 30 秒）Canvas 是即時模式的繪圖，流程是：拿到 context，每一幀先 clearRect 清空，再把所有圖元重新算、重新畫一遍，然後用 requestAnimationFrame 排下一幀。它跟 DOM 最大的差別是，DOM 有節點樹、瀏覽器會自動重繪也自帶事件；Canvas 畫完就只是像素、沒有物件，所以互動要自己做命中測試，文字排版跟狀態也都得自己管。好處是大量圖元的場景效能很高，像表格庫用 Canvas，上萬列也只畫可視區、自己畫格線跟文字，比 DOM 表格快很多。
協同編輯的衝突如何解決？（OT vs CRDT）

多人同時編輯同一份文件時，要解決並發修改衝突並維持最終一致性。兩大方案：

  • OT（Operational Transformation）：把編輯轉成 operation，透過轉換函式調整彼此 operation 的位置/索引，通常需中央伺服器協調順序。代表：Google Docs。
  • CRDT（Conflict-free Replicated Data Type）：資料結構本身保證合併無衝突，可去中心化、支援離線後再合併。代表：Yjs、Automerge。

前端實務：常用 Yjs + WebSocket / WebRTC 同步文件，並用 awareness 同步游標與選取狀態；OT 實作較複雜但省頻寬，CRDT 上手快、離線友善。

🎤 口頭講稿（約 30 秒）多人同時改一份文件，要解決並發衝突、維持最終一致。主要有兩種方案：一種是 OT，把每個編輯轉成 operation，再透過轉換函式去調整彼此的位置，通常需要一個中央伺服器來決定順序，Google Docs 就是這種；另一種是 CRDT，它的資料結構本身就保證合併不會衝突，可以去中心化、也支援離線後再合併，像 Yjs、Automerge。前端我會偏向用 Yjs 搭 WebSocket，順便用 awareness 同步游標。

AI Agent 的核心流程與底層邏輯是什麼？（不只是文字問答）

Agent ≠ 單純的聊天問答。核心是「LLM 當決策大腦 + 工具呼叫 + 記憶 + 多步驟循環」。

運作流程（ReAct：Reason + Act）：

  1. 接收任務 → LLM 規劃（planning）下一步。
  2. 決定呼叫哪個工具（function / tool calling），輸出結構化參數。
  3. 執行工具、拿到結果（observation），回饋給 LLM。
  4. 反覆循環，直到任務完成 → 產生最終回覆。

關鍵概念：System prompt、Tool/Function schema、Context window 管理、Memory（短期對話 + 長期向量檢索 RAG）、Structured output（用 Zod / JSON Schema 約束輸出）、Guardrails 與重試。

前端的角色：不只做輸入框與 streaming 渲染——還要懂 prompt 設計、tool 定義、SSE 串流、用狀態機管理 agent 的多步驟與錯誤重試。

🎤 口頭講稿（約 30 秒）Agent 跟單純的文字問答不一樣，核心是把 LLM 當決策大腦，再加上工具呼叫、記憶跟多步驟的循環。流程上大概是：LLM 先規劃下一步，決定要呼叫哪個工具、輸出結構化的參數，工具執行完把結果回饋給它，再反覆這個 Reason 加 Act 的循環，直到任務完成。重要的概念有 function calling、context 管理、用 RAG 做長期記憶，還有用 Zod 或 JSON schema 約束輸出。前端的角色其實不只是做輸入框跟 streaming，還要懂 tool 定義、SSE 串流，跟用狀態機去管 agent 的步驟跟重試。
效能優化怎麼講出「量化數據」？（被追問必備）

履歷只要寫「負責效能優化」，面試官一定追問：優化前的指標？採用的方案？優化後的效果？所以要準備「前 → 做了什麼 → 後」的數字。

常用指標：Core Web Vitals（LCP / INP / CLS）、FCP / TTI、bundle size、首屏時間、API 回應時間、長列表 FPS、記憶體用量。

講法範例：

「某列表頁初始 bundle 1.2MB、LCP 約 4.2s。我做了路由懶載入 + 圖片改 WebP + 長列表虛擬捲動，bundle 降到 480KB、LCP 縮到 1.8s，捲動 FPS 從 ~30 回到 60。」

量測工具：Lighthouse、DevTools Performance、webpack-bundle-analyzer、後端 APM。

🎤 口頭講稿（約 30 秒）我講效能優化會盡量帶數字，講「優化前、做了什麼、優化後」。舉例：某個列表頁初始 bundle 大概 1.2MB、LCP 大約 4.2 秒。我做了路由懶載入、圖片改 WebP、長列表上虛擬捲動，最後 bundle 降到 480KB、LCP 縮到 1.8 秒，捲動的 FPS 也從大概 30 回到 60。量測我會用 Lighthouse、DevTools 的 Performance 面板，還有 bundle analyzer 去找哪一包太大。

HTML

如果畫面有按鈕需求，你會使用什麼 HTML tag？

先確認這個「按鈕」的行為與語意再決定：

  • 觸發動作（送出、開關、提交）→ 用 <button>。
  • 導向另一個頁面 / 連結（有 href）→ 用 <a>。

關鍵是語意正確，而不是只看外觀；樣式可以再用 CSS 處理成一致的外觀。

為什麼要使用 HTML 語意化標籤？優點是什麼？

語意化標籤（如 <header>、<nav>、<main>、<article>）能正確表達網頁內容的含義。優點：

  • SEO：搜尋引擎更容易理解內容結構與重點。
  • 無障礙（a11y）：螢幕閱讀器能正確解讀，對輔助技術友善。
  • 好維護：結構清楚，團隊協作可讀性高。
  • 語意明確的標籤通常比一堆 <div> 更精簡。
<script> 應擺在哪？<link> 又該擺哪？

<script> 建議放在 </body> 結束前：

  • 畫面更快呈現：瀏覽器先解析完整 HTML、再下載/解析 JS，使用者能更快看到內容。
  • 避免操作到尚未載入的 DOM：放在底部可確保 HTML 元素已載入，減少 script 找不到元素的錯誤。
  • 或用 defer（解析完 HTML 後依序執行）/ async（下載完即執行）。

<link>（CSS）放在 <head> 內：確保渲染前已載入樣式，避免使用者看到「無樣式內容閃爍（FOUC）」。

CSS

CSS Specificity（優先級）怎麼算？

由高到低大致為：

!important > 行內樣式（inline style）> ID 選擇器 > class / 偽類 / 屬性選擇器 > 型態選擇器（標籤）/ 偽元素 > 通用選擇器（*）。

計算時常用 (a, b, c) 表示：a = ID 數、b = class/屬性/偽類 數、c = 標籤/偽元素 數，逐位比較。相同時後寫的覆蓋先寫的。

box-sizing 的 content-box 和 border-box 差別？預設值？
  • content-box（預設）：width 只算內容，padding 與 border 會額外撐大元素總寬。
  • border-box：width 已包含 content + padding + border，總寬就是你設定的值，排版更直覺。

實務上常全域設 * { box-sizing: border-box; }。

寫一個顏色漸變的動畫要用什麼 property？

用 transition 系列：

  • transition-property：要漸變的 CSS 屬性（如 background-color）。
  • transition-duration：持續時間（如 1000ms = 1 秒）。
  • transition-timing-function：漸變模式（ease、linear…）。

連續/循環的漸變則用 @keyframes + animation。

什麼是 Block Formatting Context (BFC)？

BFC 是一個獨立的渲染區域，內部元素的排版與浮動、清除互不影響外部。觸發 BFC（如 overflow:hidden、display:flow-root、float、絕對定位等）可以解決常見問題：

  • 清除浮動造成的高度塌陷。
  • 避免 margin 重疊（margin collapse）。
  • 避免元素被浮動元素覆蓋。
SASS 的 & 代表什麼？

& 是「父選擇器引用（Parent Selector Reference）」，用來在巢狀中引用當前選擇器，方便組合修飾。例如：

.btn {
  &:hover { color: green; }
  &--primary { background: green; }
}
/* 編譯成 .btn:hover、.btn--primary */
font-size 單位 px / em / rem / % 的差別？
  • px：絕對單位，精準但不會隨頁面/設定縮放。
  • em：相對單位，相對於父元素字級，會逐層繼承累乘，多層巢狀時容易混亂；適合簡單、少層級的區塊。
  • rem：相對於根元素（html）字級，解決了 em 的繼承問題，適合做整體可縮放的排版。
  • %：相對於父元素對應屬性的百分比。
什麼是 CSS reset？

用來統一各瀏覽器的預設樣式（margin、padding、line-height 等），確保在不同瀏覽器有一致的外觀與排版起點。常見如 reset.css、normalize.css。

Flexbox 怎麼使用？常用哪些屬性？

容器設 display:flex，常用屬性：

  • 容器：flex-direction（主軸方向）、justify-content（主軸對齊）、align-items（交叉軸對齊）、flex-wrap（換行）、gap（間距）。
  • 子項：flex（grow/shrink/basis）、align-self、order。
水平垂直置中的不同方法？
  • Flexbox：display:flex; justify-content:center; align-items:center;
  • Grid：display:grid; place-items:center;
  • 絕對定位 + transform：position:absolute; top:50%; left:50%; transform:translate(-50%,-50%);
  • 行內元素：text-align:center + line-height 等於高度。
position 有哪些屬性值？
  • static：預設，照文檔流。
  • relative：相對自身原位置偏移，仍保留原空間。
  • absolute：相對最近的「非 static 祖先」定位，脫離文檔流。
  • fixed：相對瀏覽器視窗定位，捲動不動。
  • sticky：在門檻內表現如 relative，超過門檻黏住如 fixed。
知道如何使用 mixin 與 SASS/SCSS 變數嗎？

會。變數用 $ 宣告共用值；@mixin 定義可重用樣式片段、用 @include 套用，可帶參數：

$primary: #1aa71b;
@mixin flex-center { display:flex; justify-content:center; align-items:center; }
.box { color:$primary; @include flex-center; }

JavaScript

什麼是作用域（scope）？作用域鏈？

作用域是變數可被存取的範圍，決定其可見性與生命週期。JS 有全域、函式作用域，ES6+ 還有區塊作用域（let/const）。

作用域鏈：當在函式內找變數時，會先在當前作用域找，找不到就往外層找，一路到全域為止——這條查找路徑就是作用域鏈。

什麼是 hoisting（提升）？

JS 在執行前會把宣告提升到作用域頂端。var 宣告會被提升並初始化為 undefined；函式宣告（function declaration）整個被提升、可在定義前呼叫。let/const 雖也被提升，但會進入暫時死區（TDZ），宣告前使用會報錯。

什麼是 event loop？微任務 vs 宏任務？

JS 是單執行緒，靠 event loop 處理非同步。執行順序：同步程式碼 → 清空所有微任務（microtask） → 取一個宏任務（macrotask） → 再清微任務…循環。

  • 微任務：Promise 的 then/catch、queueMicrotask、MutationObserver。
  • 宏任務：setTimeout、setInterval、I/O、UI 事件。

重點：微任務比宏任務先執行。

setTimeout、Promise、同步 console.log 的輸出順序？
  • 同步立即依序執行。
  • 微任務（Promise 回呼）在當前同步結束後立即執行，優先於宏任務。
  • 宏任務（setTimeout 回呼）在同步與微任務都清空後才執行。
console.log(1);
setTimeout(()=>console.log(2));
Promise.resolve().then(()=>console.log(3));
console.log(4);
// 輸出：1, 4, 3, 2

進階版（含 async/await，面試最愛追問）：

async function async1(){
  console.log('A');
  await async2();
  console.log('B');   // await 之後的程式碼 → 變成微任務
}
async function async2(){ console.log('C'); }

console.log('D');
setTimeout(() => console.log('E'));      // 宏任務
async1();
Promise.resolve().then(() => console.log('F')); // 微任務
console.log('G');
// 輸出：D, A, C, G, B, F, E

追蹤重點：同步先跑（D、A、C、G）→ 清微任務佇列（B、F）→ 才取宏任務（E）。await async2() 會同步執行 async2 的本體（印 C），但 await 後面的 console.log('B') 被排進微任務。

function declaration vs function expression 的差別？
  • 提升：function declaration 整個被提升，定義前可呼叫；function expression 不會（賦值前是 undefined / TDZ）。
  • 語法位置：declaration 獨立存在；expression 通常賦值給變數或作為參數傳遞。
by value vs by reference？
  • 基本型別（Number、String、Boolean…）按值傳遞，修改副本不影響原值。
  • 物件型別（Object、Array、Function）按引用傳遞，多個變數可指向同一物件，修改會影響所有引用。
不同情況下 this 的指向？
  • 一般函式呼叫：非嚴格模式指向 window/global，嚴格模式為 undefined。
  • 物件方法呼叫：指向呼叫它的物件。
  • 建構函式（new）：指向新建立的實例。
  • call/apply/bind：指向指定的物件。
  • 箭頭函式：沒有自己的 this，沿用定義時外層的 this。

看 code 講輸出（面試常考）：

const obj = { name: 'A', getName(){ return this.name; } };
const fn = obj.getName;
obj.getName();   // 'A'        ← 物件呼叫，this = obj
fn();            // undefined  ← 一般呼叫，this 非 obj
fn.call(obj);    // 'A'        ← call 指定 this

const o = {
  name: 'B',
  normal(){ setTimeout(function(){ console.log(this.name); }); }, // undefined（一般函式）
  arrow(){  setTimeout(() => console.log(this.name)); }          // 'B'（箭頭沿用外層 o）
};

手寫一個改變 this 的方法（myBind）：

Function.prototype.myBind = function(ctx, ...preset){
  const fn = this;
  return function(...args){
    return fn.apply(ctx, [...preset, ...args]);
  };
};
function say(greet){ return greet + ', ' + this.name; }
const bound = say.myBind({ name: 'Alex' });
bound('Hi'); // 'Hi, Alex'
什麼是原型鏈（prototype chain）？

原型本身也是物件、也有自己的原型。存取物件屬性時，若自身沒有，會往它的原型找，找不到再往上一層原型找，直到找到或到達 null 為止——這條路徑就是原型鏈，終點是 null。它是 JS 繼承的核心機制。

什麼是閉包（closure）？用途是什麼？

閉包是「函式 + 其被宣告時的詞法作用域環境」的組合。白話說：內部函式能存取並記住外部函式的變數，即使外部函式已執行完畢。

function outer(){
  let a = 0;
  return function inner(){ return ++a; };
}
const fn = outer();
fn(); // 1
fn(); // 2  ← a 的狀態被記住

用途：狀態保存、資料封裝（私有變數）、函式工廠、debounce/throttle 等。

Promise 熟不熟？Promise vs async/await 差別？有什麼是其中一個做得到另一個做不到的？

async/await 本質是 Promise 的語法糖，讓非同步用同步邏輯書寫、更易讀，並用 try/catch 處理錯誤。

  • Promise 更適合處理多個並發或鏈式操作（Promise.all、race、allSettled）。
  • async/await 更適合需要循序、可讀性高的流程。

嚴格說兩者能力相同（await 等的就是 Promise），主要差在表達方式與可讀性；並發控制用 Promise 的組合方法較直覺。

Promise 和 async/await 要解決的是什麼問題？

解決非同步 callback 造成的「回呼地獄（callback hell）」——深層巢狀、難讀難維護、錯誤處理複雜。Promise 用鏈式攤平流程，async/await 進一步用同步寫法表達非同步（減少 then 巢狀）。

有用遞迴方式寫過 async/await 嗎？什麼情形會用？

會用在不確定次數、需依前一次結果再決定是否繼續的場景，例如：分頁 API 一直拉到沒有下一頁、輪詢（polling）某個狀態直到完成、重試（retry）機制。每次 await 上一輪結果後，再決定要不要遞迴下一輪。

var、let、const 的差別？
  • 作用域：var 為函式/全域作用域；let、const 為區塊作用域。
  • 重複宣告：var 可；let、const 不可。
  • 提升：var 自動初始化為 undefined；let、const 進入 TDZ，宣告前使用會報錯。
  • 重新賦值：let 可；const 不可（但物件內容仍可變）。
forEach / map / filter / reduce 的差別？
  • forEach：單純遍歷執行副作用，無回傳值。
  • map：回傳等長新陣列（轉換每個元素）。
  • filter：回傳符合條件的新陣列（可能變短）。
  • reduce：把陣列累積成單一值（加總、分組、攤平…）。
如何自己實作 myMap()，功能和 map 一樣？ 
Array.prototype.myMap = function(callback, thisArg){
  const result = [];
  for(let i = 0; i < this.length; i++){
    if(i in this){ // 跳過稀疏陣列的空洞
      result.push(callback.call(thisArg, this[i], i, this));
    }
  }
  return result;
};
[1,2,3].myMap(x => x * 2); // [2,4,6]
什麼是 event bubbling？知道 bubbling 與 capture 的傳遞流程嗎？

事件冒泡（bubbling）：事件從觸發的目標元素，由內往外傳到父層、祖父層直到最頂層。

完整事件流三階段：1. 捕獲（capture）由外往內 → 2. 目標階段 → 3. 冒泡（bubble）由內往外。addEventListener 第三參數 true 表示在捕獲階段觸發，預設（false）為冒泡階段。

請解釋 callback function？優缺點？

把一個函式當參數傳入、在適當時機被呼叫，就是 callback。

優點：提高彈性與重用性、方便處理非同步、不阻塞主流程。

缺點：過度巢狀會造成回呼地獄、難讀難維護；錯誤處理需在每層各別處理，較繁瑣。

知道 debounce 嗎？debounce 與 throttle 的差異？
  • debounce（防抖）：事件停止觸發後等待 n 毫秒才執行；期間再觸發就重新計時。適合搜尋輸入、視窗 resize 結束後才動作。
  • throttle（節流）：固定時間間隔內最多執行一次。適合scroll、滑鼠移動等高頻事件限速。
什麼是淺比較和深比較？
  • 淺比較：只比第一層屬性；對巢狀物件只比引用是否相同。效率高，但可能漏掉內部變化（React 的 memo、PureComponent 就用淺比較）。
  • 深比較：遞迴比較所有層級的值，準確但效能較差。
請解釋淺拷貝與深拷貝？
  • 淺拷貝：只複製第一層；巢狀物件仍共用同一引用。如 Object.assign、展開運算子 {...obj}。
  • 深拷貝：完整複製所有層級，互不影響。如 structuredClone(obj)、JSON.parse(JSON.stringify(obj))（有限制：丟失函式、undefined、Date 變字串等）、或 lodash 的 cloneDeep。
請說出 JavaScript 所有的基本型別？

共 7 種：String、Number、BigInt、Boolean、Undefined、Null、Symbol。其餘（Object、Array、Function…）皆屬物件型別。

== 和 === 的差異？（順帶 Object.is）
  • ==（鬆散比較）：比較前會強制轉型，如 1 == '1'、null == undefined 皆為 true。
  • ===（嚴格比較）：不轉型，型別不同直接 false。例外：NaN === NaN 為 false、+0 === -0 為 true。
  • Object.is（同值比較）：修正上述兩個例外——Object.is(NaN, NaN) 為 true、Object.is(+0, -0) 為 false。判斷 NaN 也可用 Number.isNaN。
使用過什麼方法處理 AJAX？

主要用 Axios 與原生 Fetch API。Axios 提供攔截器、自動轉 JSON、取消請求等便利功能；簡單情境用 Fetch。

普通函式與箭頭函式的區別？
  • this：箭頭函式沒有自己的 this，沿用外層；普通函式的 this 視呼叫方式而定。
  • arguments：箭頭函式沒有 arguments（可用 rest 參數）。
  • 不能當建構函式：箭頭函式不能 new，也沒有 prototype。
  • 不能用作 generator（無 yield）。
同步與非同步的差異？
  • 同步：程式碼依序執行，會阻塞後續操作直到完成。
  • 非同步：可在等待操作完成的同時繼續執行，不阻塞，完成後再以 callback/Promise 回來處理。

TypeScript

any vs unknown 的差別？
  • any：完全關閉型別檢查，可任意操作，降低型別安全。
  • unknown：更安全的 any——可以接收任何值，但使用前必須先縮小型別（type guard / 斷言），否則無法直接操作。

實務上不確定型別時優先用 unknown，逼自己先檢查。

type 跟 interface 的差別？
  • interface：可被重複宣告自動合併（declaration merging）、可被 extends，較適合描述物件／類別的形狀。
  • type：更通用，可定義聯合型別、交集、元組、原始型別別名、條件型別等，不可重複宣告合併。

實務：描述物件/可被擴充的公開 API 偏好 interface；需要聯合、工具型別等彈性時用 type。

請說明泛型（generics）是什麼？

泛型是「型別的參數化」，讓函式/型別能適用多種型別而不重複程式碼，同時保留型別安全。

function identity<T>(value: T): T { return value; }
identity<string>("hi"); // 型別被保留為 string

好處：彈性、型別安全、可讀性（清楚表達輸入輸出的型別關係）。

extends 在泛型中的用途？

extends 用來對型別參數施加約束，確保傳入型別滿足特定條件，提升安全與更好的提示。

function getLength<T extends { length: number }>(x: T){ return x.length; }
getLength("abc"); // OK
getLength([1,2]);  // OK
getLength(123);    // ❌ number 沒有 length
enum 和 const 的差別？
  • enum：TS 特有，定義一組具名常數（數字型預設從 0 遞增，或字串型），可雙向對應，可讀性好。
  • const：JS 的常數宣告，用於單一常數或物件，不會自動產生具名集合。

實務上很多人改用 as const 物件 + union 取代 enum，避免 enum 編譯產物與 tree-shaking 問題。

JS 與 TS 的區別？

TS 是 JS 的超集，加上靜態型別系統（編譯期檢查）、介面、泛型、列舉等。最終會編譯成 JS 才能在瀏覽器/Node 執行。好處是把許多錯誤提前到編譯期、提升可維護性與工具提示。

（實戰）你怎麼用 Zod / ts-rest 做型別安全？

Zod 是執行期 schema 驗證庫，能用 z.infer 從 schema 直接推導出 TS 型別——做到「一份 schema，同時管 runtime 驗證與靜態型別」。

ts-rest 則把 API contract 定義成共享的型別來源，前後端都引用同一份合約：後端 schema 一改，前端在編譯期就會報錯。我在現職以 ts-rest + Zod 推行 Contract-first，串接錯誤明顯減少。

React

React component 的生命週期？

Class：掛載 constructor → render → componentDidMount；更新 render → componentDidUpdate；卸載 componentWillUnmount。

Function（Hooks）對應：useEffect(fn, []) ≈ didMount；useEffect(fn, [dep]) ≈ didUpdate；effect 內 return 的清理函式 ≈ willUnmount。

哪些情況會造成 component re-render？
  • state 改變（setState）。
  • props 改變。
  • 父元件重新渲染（子元件預設跟著渲染，除非 memo）。
  • 使用的 Context 值改變（所有消費該 Context 的元件都會重渲染）。
React key 的用途？

key 讓 React 在 diff 時辨識哪些子元素被新增、移除或移動，避免與狀態相關的 bug，並提升更新效率。應使用穩定且唯一的值（如資料 id），避免用 index（列表會增刪/排序時易出錯）。

用過 React Hook 嗎？它要解決什麼問題？

用過。Hooks 解決了 class component 的幾個痛點：

  • 狀態相關邏輯在 class 間難以複用（以前靠 HOC / render props，易巢狀）→ 用 custom hook 抽離複用。
  • 複雜元件邏輯散落在各生命週期，難理解 → 用 effect 依關注點組織。
  • class 的 this 對開發者不直覺。
React Hook 的規則有哪些？
  • 只能在最頂層呼叫：不能放在 if/for/巢狀或 return 之後（要確保每次 render 的 Hook 呼叫順序一致）。
  • 只能在 React 函式中呼叫：function component 或 custom hook 裡，不能在一般 JS 函式中用。
useState 給初始值 vs 給 function（惰性初始化）差別？
  • 直接初始值：適合簡單、靜態的初始狀態。
  • 函式（惰性初始化）：useState(() => expensiveCompute()) 只在首次 render 執行一次，適合初始值需要昂貴計算時，避免每次 render 都重算。
useState / setState 要注意什麼？是同步還是非同步？

setState 是非同步（批次更新）的——呼叫後不會立即拿到新值。注意事項：

  • 需依前一狀態更新時，用函式形式：setCount(c => c + 1)。
  • 不可直接修改 state 物件/陣列，要建立新的（展開運算子）。
  • 同一輪多次 setState 會被合併批次處理。
useState 放巢狀資料（array of objects）更新時要注意什麼？
  • 不可直接改原陣列/物件屬性。
  • 用展開運算子建立新的陣列/物件，保留未變動的部分（保持 immutable）。
  • 依舊狀態更新時用函式形式 setState。
setList(prev => prev.map(item =>
  item.id === id ? { ...item, done: true } : item
));
useEffect 怎麼用？第二個參數的作用？回傳的函式用途？

useEffect 處理副作用（請求、訂閱、DOM 操作）。第二參數 dependency array：

  • 空陣列 []：只在掛載時執行一次（≈ componentDidMount）。
  • 有依賴 [dep]：dep 改變時重新執行（≈ componentDidUpdate）。
  • 不傳：每次 render 都執行。

return 的清理函式：在元件卸載前、或下次 effect 執行前呼叫，用來清理訂閱/計時器，避免記憶體洩漏。

useEffect 的 dependency array 放物件或陣列會怎樣？

依賴比較用的是引用（reference）。物件/陣列每次 render 若是新建立的，引用就不同，會讓 effect 每次都觸發；反之若引用沒變、即使內容變了也不會觸發。解法：用 useMemo 穩定引用，或在依賴中放原始值而非整個物件。

用過 useLayoutEffect 嗎？跟 useEffect 差在哪？

兩者參數相同，差在執行時機：useLayoutEffect 在瀏覽器重繪（paint）之前同步執行，適合需要量測 DOM 或避免畫面閃爍的情況；useEffect 在 paint 之後非同步執行，是預設選擇（不阻塞繪製）。

useRef 怎麼用？跟 useState 差別？
  • useRef：回傳可變的 .current，改變不會觸發重渲染，跨 render 保持值。常用於取 DOM 節點、存計時器 id、保存不需呈現的可變值。
  • useState：改變會觸發重渲染，用於要反映在畫面上的資料。
useRef 的值改變會觸發 useEffect 嗎？放到 component 之外呢？

不會。改 ref.current 不會觸發重渲染，因此 effect 也不會因它而執行（React 偵測不到 ref 變化）。把變數放在 component 外（模組層級全域變數）同樣不會被 React 追蹤、也不會觸發 effect。要驅動畫面/effect 請用 state。

memo 跟 useMemo 差別？useCallback vs useMemo？
  • React.memo：包元件，props 淺比較未變就跳過重渲染。
  • useMemo：快取計算結果，依賴未變就不重算，避免昂貴運算或穩定物件引用。
  • useCallback：快取函式引用（useCallback(fn, deps) ≈ useMemo(() => fn, deps)），常用於傳給 memo 子元件的 callback。
把所有變數都用 useMemo 包起來可以嗎？

不建議。useMemo 本身有記憶與比較成本，對簡單或經常變動的值反而得不償失，也讓程式碼變複雜。最佳實踐：只在計算昂貴、或需要穩定引用避免子元件重渲染時才用，保持可讀性優先。

下層元件的 state 要給上層用，怎麼做？

常見做法：

  • 狀態提升 + 回呼：把 state 放到上層，傳 callback 給下層，由下層呼叫回傳資料（最直接）。
  • Context API：需要跨多層共享時。
  • 狀態管理庫：Redux/Zustand/Pinia 等，適合大型應用。
聽過 React batch update 嗎？

React 會把同一事件中的多次 state 更新合併成一次重渲染，減少不必要的渲染、提升效能。React 18 起，連 setTimeout、Promise 等非同步情境也自動批次（automatic batching）。

什麼是 prop drilling？如何避免？

prop drilling 指資料需要層層往下傳經過許多中間元件（即使中間元件用不到），造成維護困難。避免方式：Context API、狀態管理庫、或重構元件結構（component composition、children）。

Context 和 Redux 的差別？可以用 Context 取代 Redux 嗎？

Context 是「依賴注入 / 傳值」機制，適合低頻變動的全域資料（主題、語系、登入者）。Redux 是完整狀態管理，提供可預測的更新流程、middleware、devtools、時間旅行除錯，適合大型、高頻、複雜的狀態。

簡單情境可用 Context 取代 Redux；但 Context 值一變，所有消費者都重渲染，高頻更新時效能與可維護性不如 Redux（或 Zustand / Jotai）。

要寫出好維護的 React code，需要注意哪些？

遵循 SOLID 原則的精神：

  • S 單一職責：元件/hook 只做一件事，過大就拆。
  • O 開放封閉：用 props/composition 擴充，而非改既有元件。
  • L 里氏替換、I 介面隔離、D 依賴反轉。

實務：抽共用邏輯成 custom hook、保持 immutable、合理 memo、明確的型別與資料流、避免 prop drilling。

用過 Redux 嗎？它要解決什麼問題？

用過（含 Redux Toolkit）。Redux 解決大型應用的狀態管理複雜性與跨元件/跨層級通訊——不必把資料一層層往下傳。它提供單一資料來源、可預測的單向資料流（action → reducer → store），便於測試與除錯。代價是有些樣板程式碼（RTK 已大幅簡化）。

Vue

（實戰）Options API 重構成 Composition API 的經驗？

我主導過把舊有 Options API 重構為 Composition API。Composition API 用 setup / <script setup> 把同一個關注點的邏輯聚在一起（不再散落在 data/methods/computed），並能抽成可複用的 composables（use 開頭函式）。重構後程式碼更清晰、邏輯複用更容易，團隊維護成本明顯下降。

雙向數據綁定的原理？

Vue 2：用 Object.defineProperty 對 data 的每個屬性設 getter/setter，做依賴收集與派發更新。

Vue 3：改用 Proxy 代理整個物件，能攔截屬性的新增/刪除、支援陣列與巢狀，效能與覆蓋度都更好。v-model 本質是 :value + @input 的語法糖。

Vue 2 給物件新增屬性會發生什麼？如何解決？$set 原理？

Vue 2 因為用 Object.defineProperty 在初始化時才設 getter/setter，後加的新屬性不是響應式，畫面不會更新。解法：用 Vue.set(obj, key, value) / this.$set(...)，它會對新屬性補上響應式定義並觸發更新。Vue 3 用 Proxy 後此問題不存在。

Vue 打包優化方案？
  • 路由懶載入（動態 import）+ 程式碼分割。
  • 元件按需載入、第三方 UI 庫 tree-shaking。
  • 圖片壓縮、資源 CDN。
  • 開啟 Gzip/Brotli 壓縮。
  • 用 Vite/Rollup 做 build 分析（visualizer）找出大包。
  • 避免重複打包（externals / 共用 chunk）。
引用 Pinia 資料時前面的 ... 是什麼意思？

那是展開運算子，搭配 storeToRefs(store) 或 mapState 把 store 的多個 state/getter 一次展開到元件中使用。注意：解構 store 會失去響應式，所以要用 storeToRefs 保留響應性；而 actions（方法）則可直接解構。

Vuex 與 Pinia 的區別？
  • 更簡潔：Pinia 沒有 mutations，直接在 action 改 state；API 更貼近 Composition。
  • TypeScript 友善：Pinia 型別推導開箱即用。
  • 模組化：Pinia 天生多 store、扁平，不需 Vuex 的 modules 巢狀。
  • Pinia 是 Vue 官方推薦、Vue 3 的預設狀態管理方案。
Vue 怎麼做權限管理？
  • 路由層：路由 meta 標記所需權限，搭配全域路由守衛（beforeEach）判斷是否放行；登入後依角色動態產生可存取路由（addRoute）。
  • 畫面層：用自訂指令 v-permission 或條件渲染控制按鈕/選單顯示。
  • 後端把關：前端權限只是體驗，真正權限仍要後端驗證。

框架共通 / 選型

React 和 Vue 的區別？
  • 模板：React 用 JSX（JS 為主）；Vue 用模板語法（也支援 JSX）。
  • 響應式：React 靠 setState + 重渲染 + diff；Vue 靠 Proxy/getter-setter 的細粒度響應式追蹤。
  • 心智模型：React「一切是函式、手動最佳化（memo）」；Vue 自動追蹤依賴、API 較規範。
  • 兩者都用 Virtual DOM、元件化、單向資料流（Vue 的 v-model 是雙向語法糖）。

我兩者都實際主導過專案，能依團隊與場景選型。

為什麼選擇你現在用的框架？

主要是生態圈完善、社群與人才多，遇到問題容易找到解法與資源；UI 庫、狀態管理、工具鏈都成熟。實務上我會依團隊既有技術棧與專案需求選擇——現職車隊系統用 Vue 3，先前後台多用 React。

用過其他 JS 框架嗎？跟現在用的差別？

用過 Vue（2 與 3）、React，以及一點 Angular。最大差別在於模板與心智模型：React 以 JSX/函式為核心、需手動最佳化；Vue 模板語法直覺、響應式自動追蹤；Angular 較完整框架、TypeScript 與 DI 為核心、學習曲線較陡。

<Link> 和 <a> 有什麼區別？

<Link> 是 react-router-dom（Vue 是 <router-link>）的元件，頁面無刷新跳轉（SPA）；<a> 是原生標籤，會整頁刷新重載。<Link> 渲染後其實也是 <a>，但內部用 History API 跳轉並 preventDefault() 攔截了原生行為，避免刷新。

有用過 Webpack 嗎？它解決什麼？

用過。Webpack 是模組打包工具：把 JS/CSS/圖片等各種資源視為模組，依依賴關係打包，並透過 loader（轉譯，如 babel/sass-loader）與 plugin（額外功能，如壓縮、HMR）處理。解決了瀏覽器模組化、相依管理、轉譯與最佳化（code splitting、tree-shaking）等問題。

Vite 是什麼？為什麼要用？跟 Webpack 的區別？
  • 開發伺服器：Vite 用瀏覽器原生 ES Module + esbuild，啟動幾乎即時、HMR 快；Webpack 需先打包整包，專案大時啟動慢。
  • 生產建置：Vite 用 Rollup 打包。
  • 設定：Vite 開箱即用、設定精簡；Webpack 彈性大但設定繁瑣。

它解決了大型專案 dev server 啟動與熱更新慢的痛點。

平常寫 code 有用 linter 嗎？用哪個？

有。用 ESLint 檢查潛在 bug 與規範、Prettier 統一格式，並搭配 Husky 在 commit 前自動跑（lint-staged）。簡記：Prettier 管風格、ESLint 抓問題。

什麼是 SPA？優缺點？

Single-Page Application：整個應用在單一頁面內以 JS 動態切換內容、不整頁重載。

優點：體驗流暢、前後端分離、切頁快。

缺點：首屏 JS 大、首次載入較慢；SEO 較差（解法見 SSR / 預渲染）。

什麼是 SSR？怎麼解決 SPA 的 SEO 與首屏問題？

SSR（伺服器端渲染）在伺服器產生完整 HTML 再送給瀏覽器，讓搜尋引擎能正確索引、首屏更快，之後再 hydration 接手互動（SSR + CSR）。框架如 Next.js（React）、Nuxt（Vue）。

SPA 的 JS 過大問題則用懶載入、code splitting、壓縮等技術改善。

Canvas 和 WebGL 的差異？有用過 Three.js 嗎？

Canvas 適合 2D 繪圖與簡單動畫；WebGL 是基於 GPU 的高效能 3D 渲染 API。Three.js 是封裝 WebGL 的 3D 函式庫，降低門檻。我目前在學 Three.js → React Three Fiber (R3F) → Drei，應用場景含 3D 資料視覺化、車隊路徑/倉儲視覺化、VR/AR、互動裝置。

知道 WebRTC 嗎？

WebRTC（Web Real-Time Communication）是開源技術，讓瀏覽器/App 點對點即時傳輸音訊、視訊與資料，不需外掛。應用：視訊會議（Google Meet/Zoom）、語音通話、即時客服、線上遊戲資料交換、遠距教學。

知道 WebAssembly 嗎？

WebAssembly（Wasm）是一種可在瀏覽器執行的低階位元組碼，讓 C/C++/Rust 等語言編譯後在 Web 上接近原生效能執行。適用：遊戲、影像/影音編解碼、科學計算、大量運算的資料處理，以及把既有原生應用移植到 Web。

styled-components vs CSS Modules？
  • styled-components（CSS-in-JS）：樣式與邏輯緊密結合、支援動態樣式（依 props）、元件化開發。
  • CSS Modules：樣式與邏輯清楚分離、用傳統 CSS、自動作用域避免命名衝突、無 runtime 成本。
知道 Tailwind 嗎？優缺點？

優點：utility class 快速開發、設計一致、可由設定檔高度客製、易維護（樣式跟結構在一起）、內建 RWD 工具。

缺點：初學有學習曲線、class 名冗長影響可讀性、需建置工具支援；複雜元件仍需自行抽象。

我在多個專案實際使用，後台與 POC 介面開發效率明顯提升。

有用過哪些 CSS 框架 / UI 庫？

CSS 框架：Bootstrap、Tailwind CSS。UI 元件庫：Ant Design 3/4/5（含 ProComponents）、Element UI / Element Plus、Mint UI。

什麼是 Monorepo？你怎麼用？

Monorepo 是把多個關聯專案/套件放在同一個 repo 管理（用 pnpm workspace / Turborepo / Nx 等）。好處：共用型別與工具鏈、跨套件原子化提交、依賴一致。我在現職的車隊系統用 Monorepo 與 NestJS 後端共構，前後端共享 ts-rest 合約與型別，能同步迭代、不再因介面定義反覆對齊。

版本控制（Git）

請說明你使用 Git 版控的流程？

開分支（feature / fix）→ 開發提交 → rebase 同步主幹、整理 commit → 開 PR → code review → merge。採 Git-flow 概念，commit 訊息語意化。

最近在 Git 上學到什麼？rebase 與 squash？

更熟練 git rebase（讓分支基於最新主幹、線性歷史）與壓縮 commit（squash）——把多個瑣碎 commit 合併成一個有意義的提交，讓歷史更乾淨、review 更好讀。也注意 rebase 不要對已共享的公開分支使用，避免改寫他人歷史。

網路與瀏覽器

有實作過登入功能嗎？怎麼做的？

有。流程大致：表單送出帳密 → 後端驗證後回傳 token（JWT） → 前端儲存（HttpOnly Cookie 較安全，或視情況 localStorage）→ 後續請求帶上 token（Axios 攔截器統一加 Authorization header）→ 路由守衛 / ProtectedRoute 控制未登入導向；token 過期用 refresh token 換新。前端權限只是體驗，最終仍由後端把關。

Cookie、localStorage、sessionStorage 的差別？
  • Cookie：容量小（~4KB），會自動隨 HTTP 請求夾帶，適合身分辨識；可設 Expires/Max-Age，預設關閉瀏覽器失效。
  • localStorage：容量大（~5MB），永久保存直到手動/程式清除；適合長期、跨頁資料。
  • sessionStorage：容量同 local，關閉分頁/瀏覽器即清除；適合單次工作階段的暫存（如表單暫存）。

local/session 不會自動隨請求送出。

知道哪些 HTTP methods？

常見：GET（查詢）、POST（新增/提交）、PUT（整體替換）、PATCH（部分更新）、DELETE（刪除）。

GET 與 POST 的差異？
  • 用途：GET 取得資料；POST 提交/建立資料。
  • 傳輸：GET 參數在 URL（可見、有長度限制）；POST 在 request body（適合大量/敏感資料）。
  • 安全/快取：GET 可被快取、留在歷史紀錄；POST 不會。
  • 冪等性：GET 冪等、無副作用、可安全重試；POST 非冪等，重送可能造成重複資料。
PUT 與 PATCH 的差異？
  • PUT：整體替換資源，需提供完整欄位；冪等。
  • PATCH：部分更新，只傳要改的欄位；是否冪等視實作而定。
什麼是 CORS？如何解決跨域問題？

瀏覽器的同源政策會阻擋不同源（協定/網域/埠不同）的資源存取。CORS（跨來源資源共用）是讓伺服器透過回應 HTTP header（如 Access-Control-Allow-Origin）告訴瀏覽器「允許某來源存取」。

解法：最直接是請後端設定 CORS header；開發階段也可用 proxy（dev server 代理）繞過；或同源部署。

HTTP 狀態碼 304 代表什麼？是好還是不好？

304 Not Modified 表示資源未變更，瀏覽器可直接用快取，伺服器不需重傳內容。這是良好的效能最佳化，能減少頻寬與載入時間，通常是好事。

請解釋事件委派（delegation）、捕獲、冒泡？

事件流：捕獲（外→內）→ 目標 → 冒泡（內→外）。

事件委派利用冒泡：把監聽器掛在共同父層，靠 event.target 判斷實際觸發的子元素。好處：少綁很多監聽器、動態新增的子元素也能被處理、省記憶體。

效能優化

如何加快頁面載入速度？（總覽）
  • 減少請求：合併檔案、用 sprite/icon font、移除不必要資源。
  • 壓縮：開啟 Gzip/Brotli；壓縮 JS/CSS/圖片（用 WebP、適配尺寸）。
  • 快取：瀏覽器快取（Cache-Control）、CDN、Service Worker。
  • 延後載入：圖片/元件 lazy loading、JS code splitting、按需載入。
  • 資源順序：CSS 放 <head>、JS 放底部或用 async/defer。
  • 協定：HTTP/2、HTTP/3 多路複用。
  • 預載：<link rel="preload"> 關鍵資源。
  • 減少 DOM 複雜度、降低 reflow/repaint。
後端有一大包資料（如 5MB 文字），如何縮短使用者等待？
  • 分頁（pagination）：分批請求，先顯示部分。
  • 懶載入 / 無限捲動：用 IntersectionObserver 按需載入。
  • 串流（streaming）：用 fetch 的 response.body.getReader() 邊收邊處理。
  • 壓縮：後端 Gzip 傳輸，前端解壓。
  • 虛擬列表（virtualization）：只渲染可視範圍。
  • 快取 / 背景預載 / 顯示進度條降低等待焦慮。
  • 與後端協商精簡資料結構或用更緊湊格式。
CSS 放前面、JS 放後面對效能的影響？

CSS 放 <head>：渲染前先備妥樣式，避免無樣式內容閃爍（FOUC），讓畫面正確、快速呈現。

JS 放 </body> 前：JS 載入/執行會阻塞 HTML 解析，放底部可讓內容先顯示；或用 async（下載完即執行）/ defer（HTML 解析完依序執行）。

本機載入 JS 與 CDN 載入的差異？
  • 本機：穩定可控、易除錯與版本管理；但跨地區使用者載入可能較慢、無法共享快取。
  • CDN：全球節點就近載入、頻寬大、熱門庫可能已被使用者快取（共享）；但依賴第三方（中斷/被封鎖風險）、控制與隱私需留意。
打包成單一檔 vs 分開載入的差異與效能影響？
  • 打包單檔：請求數少、利於快取與壓縮（tree-shaking）；但首次需下載整包，首屏較慢，更新後整包失效。
  • 分開/按需載入：透過 code splitting + lazy loading，只載入當前所需，首屏快；但請求數增加、快取管理較細。

實務上兩者結合：核心打包、其餘按路由/功能分割懶載入。

遇到網頁效能問題，你有什麼策略？
  1. 量測：用 Chrome DevTools / Lighthouse / WebPageTest，看 FCP、LCP、FID/INP、CLS。
  2. 定位瓶頸：是網路、JS 執行、渲染（reflow/repaint）還是資源過大。
  3. 對症優化：資源壓縮、快取、懶載入、減少請求、最佳化關鍵 CSS、減少重繪重排、用 CSS 動畫取代 JS 動畫。
  4. 持續驗證：A/B 測試、定期 review，持續迭代。
如何減少 reflow（重排）與 repaint（重繪）？
  • 批次 DOM 操作：用 DocumentFragment、一次性更新，避免在迴圈中讀寫交錯。
  • 避免逐項改 style：改用 class 切換。
  • 用 transform / opacity 做動畫（走合成層，不觸發 reflow）。
  • 避免頻繁讀取會強制同步佈局的屬性（offsetWidth 等）。

設計方法 / 設計模式

什麼是 OOP？

物件導向程式設計，把現實實體建模成物件（屬性 + 方法）。四大特性：封裝、繼承、多型、抽象。優點是結構清晰、易維護與擴充，適合複雜系統。例如動物園系統用 Animal 基底類別，Lion、Elephant 繼承並覆寫各自行為。

什麼是 MVC？優勢？

MVC = Model（資料/邏輯）、View（畫面）、Controller（協調輸入與更新）。把應用分層解耦，各司其職。優勢：關注點分離、易測試與維護、利於分工。

聽過 functional programming 嗎？

函式式程式設計強調純函式（相同輸入恆得相同輸出、無副作用）、不可變資料、與函式組合（高階函式）。好處是程式更可預測、易測試與除錯。例如 add、multiply 都是純函式，再用高階函式組合成複雜運算。React 的純元件與 Redux reducer 都受 FP 影響。

什麼是 immutable / immutability？

immutable 指物件建立後不可被改變，要更新就建立新副本而非改原值。好處：狀態可預測、容易做淺比較（React/Redux 依此判斷是否更新）、避免意外副作用。實作上用展開運算子、Object.freeze、或 Immer 等。

資料結構與演算法

解釋 Stack 與 Queue，差別與應用場景？
  • Stack（堆疊）：後進先出（LIFO）。應用：函式呼叫堆疊、瀏覽器上一頁、復原（undo）、運算式求值。
  • Queue（佇列）：先進先出（FIFO）。應用：任務排程、訊息佇列、BFS、請求排隊處理。
字串處理 / 型別轉換常見題

常見可準備：反轉字串、判斷回文、字元計數、去重、找最長不重複子字串等；型別轉換注意 Number() / parseInt() 差異、隱式轉型（+ 與字串、truthy/falsy）、String() 與模板字串。建議實際手寫幾題並能說明時間複雜度。
